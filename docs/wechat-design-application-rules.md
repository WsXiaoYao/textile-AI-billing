# 微信小程序设计指南应用规则

整理日期：2026-04-27

来源：微信小程序设计指南  
https://developers.weixin.qq.com/miniprogram/design/

这份文档把微信官方设计指南转成“纺织开单小工具”的实际设计规则。后续实现页面时，先按这里检查，再对照 Figma 还原。

颜色、字号、间距、按钮、卡片等基础 token 见 `docs/design-foundations.md`。

## 1. 总原则

官方指南的四个核心方向，应用到本项目后是：

- 友好礼貌：每个页面只突出一个主任务，减少装饰和无关入口。
- 清晰明确：用户始终知道当前在哪、下一步做什么、怎么返回。
- 便捷优雅：少打字，多选择、多识别、多复用历史数据。
- 统一稳定：客户、商品、金额、按钮、反馈在所有页面保持同一套样式和交互。

## 2. 页面重点

每个页面必须只有一个主要目标。

开单首页：

- 主目标：帮销售快速完成“输入开单内容 -> 识别 -> 确认购物车”。
- 首屏重点：当前客户、输入/识别入口、识别结果、购物车摘要。
- 不放大面积品牌介绍、营销文案、无关入口。

购物车确认页：

- 主目标：确认商品、数量、金额，并进入生成订单。
- 重点信息：商品名、规格/颜色、数量、单价/金额、合计。
- 删除、修改数量、提交订单都要有明确反馈。

订单页：

- 主目标：快速查单和看状态。
- 重点信息：客户、订单号、时间、金额、收款/欠款状态。
- 列表必须能扫读，不做过度卡片化装饰。

客户页：

- 主目标：找客户、看欠款、进入开单或历史订单。
- 重点信息：客户名称、标签、欠款、最近订单。

## 3. 流程规则

官方指南强调流程明确，不能在用户目标流程中插入无关内容。本项目执行为：

1. 开单链路固定为：选客户 -> 输入 -> 识别 -> 确认 -> 生成订单 -> 收款/欠款。
2. 识别结果不能自动覆盖用户已确认的购物车内容，必须可撤销或可修改。
3. 生成订单前必须展示合计金额和客户信息。
4. 错误、缺货、规格无法识别时，在当前区域提示，不把用户跳走。
5. 次要操作放到更多菜单或二级页，不挤占主链路。

## 4. 导航规则

小程序内不绘制系统层：

- 不画手机时间、电量、信号。
- 不画微信右上角胶囊。
- 不画底部 Home Indicator。

导航使用规则：

- 默认使用微信原生导航栏。
- 只有强业务理由才用 `navigationStyle: custom`。
- tabBar 用微信原生 `tabBar`，不手写固定底栏。
- tab 建议控制在 4 个以内：开单、订单、客户、我的。
- 详情页、编辑页、确认页从 tab 页进入，返回路径要清楚。

胶囊避让：

- 页面右上角不要放主要操作。
- 如果顶部业务区有按钮，必须避开胶囊区域。
- Figma 中靠近胶囊的“更多、购物车、重置”等操作，落地时要重新计算可点击区域。

## 5. 反馈规则

官方指南建议减少等待、反馈及时。本项目执行为：

识别中：

- 输入框或识别按钮进入 loading 状态。
- 不使用全屏遮罩，除非页面完全不可操作。
- 识别超过 2 秒时显示明确文案，例如“正在识别商品和数量”。

识别成功：

- 在识别结果区域直接显示商品行。
- 给出“已加入购物车”或“待确认”的局部状态。

识别失败：

- 说明失败原因，例如“没有识别到数量”。
- 提供继续编辑或手动添加，不让用户卡住。

提交订单：

- 提交按钮进入 loading 状态。
- 成功后进入订单结果或订单详情。
- 失败时保留当前购物车内容。

加载限制：

- 同一页面最多一个主要 loading。
- 能局部 loading 就不用全局 loading。
- 不用没有说明的转圈占位。

## 6. 输入规则

官方指南强调减少输入。本项目执行为：

- 客户用搜索、最近客户、标签切换，不让用户每次完整输入。
- 商品用自然语言识别、历史商品、规格建议，不依赖纯键盘。
- 数量、单位、颜色尽量从识别结果中拆出，并允许点选修改。
- 常见单位如“米、卷、件、码”要成为可选项。
- 常用规格如“25码寸布、280祥云、H513-米”要能复用。

## 7. 触控规则

官方指南提到移动端点击热区要避免误操作。本项目执行为：

- 主要按钮高度不低于 80rpx。
- 紧邻的按钮之间至少保留 16rpx 间距。
- 列表行可点击区域覆盖整行，不只点文字。
- 删除、清空、提交等高风险操作不要贴得太近。
- 商品数量加减、客户切换、购物车入口必须容易点中。

## 8. 字体与信息层级

微信内字体跟随系统字体。本项目建立以下层级：

- 页面标题：34rpx 到 40rpx，少用。
- 模块标题：30rpx 到 34rpx。
- 正文：28rpx 到 30rpx。
- 辅助信息：24rpx 到 26rpx。
- 金额重点：40rpx 到 48rpx，但只用于应收、合计、欠款。

业务信息优先级：

1. 客户名。
2. 商品名。
3. 规格/颜色。
4. 数量。
5. 金额。
6. 状态和备注。

文字不得互相挤压：

- 长客户名允许换行或省略，但不能压住金额。
- 金额区域固定宽度或独立成列。
- 商品规格过长时换行，不挤掉数量。

## 9. 色彩规则

色彩要服务业务状态，不做单纯装饰。

- 主蓝：主按钮、选中态、可点击链接。
- 绿色：识别成功、可继续、已加入。
- 红色：欠款、错误、删除、危险操作。
- 黄色/橙色：待确认、警告、部分识别。
- 灰色：辅助说明、边框、禁用状态、页面背景。

限制：

- 不用大面积强渐变。
- 不让蓝色淹没所有信息。
- 金额红色只用于需要强调的应收、欠款、错误，不用于普通装饰。

## 10. 按钮规则

按钮必须表达清楚动作等级。

- 主按钮：一页一个主要动作，如“识别”“提交订单”。
- 次按钮：重置会话、手动添加、查看详情。
- 文本按钮：编辑、删除、更多。
- 危险按钮：删除、清空、作废，必须弱化默认曝光或二次确认。

按钮文案：

- 使用动词开头。
- 避免“确定”这种不说明动作的文案。
- 例：用“提交订单”，不用“确定”；用“清空购物车”，不用“清空”。

## 11. 图标规则

图标只作为辅助识别，不替代关键文字。

- tabBar 图标统一线性风格。
- 页面内工具图标统一线宽和圆角。
- 重要操作要有文字，不能只有图标。
- 删除、提交、购物车、搜索、设置等图标全项目统一。
- tabBar 图标按微信规则使用本地 PNG，建议 81px * 81px，单个小于 40KB。

## 12. 组件应用

优先复用这些小程序式组件模式：

- 客户卡：客户名、标签、欠款/合同金额。
- 列表行：左主信息、右状态或金额。
- 表单行：标签、输入/选择、错误提示。
- 局部反馈：识别中、识别失败、已加入购物车。
- 底部操作区：输入框和主按钮。

组件不应嵌套过深：

- 页面分区不要都做成大卡片。
- 卡片只用于真实对象，如客户、订单、商品。
- 工具区和页面背景保持简洁。

## 13. 落地检查清单

每个页面实现后检查：

1. 是否剥离了手机系统 UI 和假胶囊。
2. 页面是否只有一个主任务。
3. 主要按钮是否足够明显且只有一个。
4. 用户能否知道当前在哪、下一步做什么、怎么返回。
5. 是否减少了手动输入。
6. loading、成功、失败、空态是否完整。
7. 点击区域是否足够大。
8. 长客户名、长商品名、金额是否会挤压。
9. tab、图标、按钮是否统一。
10. 页面是否比 Figma 截图更像真实小程序，而不是手机壳截图。
