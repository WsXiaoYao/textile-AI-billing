# 微信小程序落地规则草案

## 当前阶段

已进入首个页面开发阶段。当前工程启用原生 tabBar，先落地“首页 / 开单”练手页，其余 tab 页保持占位。仍然不保留临时 mock 业务路由、不手写底栏、不自绘手机头部。

详细资料总结见 `docs/wechat-miniprogram-research.md`。微信官方设计指南的项目应用规则见 `docs/wechat-design-application-rules.md`。颜色、字号、间距、按钮、卡片、图标等基础规范见 `docs/design-foundations.md`。后续页面落地以前，先按这些资料确认工程结构、组件、WXSS、图标和设计规范。

## 开发原则

1. 先定信息架构，再写页面。
2. 先定视觉规则，再写 WXSS。
3. 先定数据模型，再写交互逻辑。
4. 小程序内不绘制 Figma 里的模拟手机状态栏、胶囊按钮、时间、电量。
5. 默认使用微信原生导航栏；只有产品确实需要沉浸式顶部时，才讨论 `navigationStyle: custom`。
6. tab 页使用原生 `tabBar`；跳转 tab 页用 `wx.switchTab`，非 tab 详情页用 `wx.navigateTo`。
7. 页面布局用 `rpx`、弹性布局、明确的宽高约束，避免靠截图目测堆位置。
8. 不把工具脚本、截图产物、文档打进小程序包。
9. 默认检查只做静态检查，不自动控制微信开发者工具 GUI。
10. 每次落地一个页面前，先写清楚该页面的业务目标、状态、数据、空态、错误态和交互路径。

## 设计还原规则

1. Figma 中属于手机外壳/系统 UI 的区域不落地。
2. Figma 中属于小程序真实业务 UI 的区域才落地。
3. 每个页面先拆成模块：导航、主内容、操作区、反馈区、弹窗/浮层。
4. 同类模块抽象为可复用样式或组件，但不要过早抽象。
5. 卡片圆角默认不超过 8px，除非 Figma 或业务组件明确要求。
6. 文案必须适配小屏幕，不允许按钮文字、金额、客户名互相挤压。
7. 金额、数量、客户名、商品名这些高频业务信息优先保证可读性。
8. 设计以微信官方设计指南为上位规范：重点突出、流程明确、导航清晰、反馈及时、减少输入、避免误操作、统一稳定。
9. 页面要像真实微信小程序，不像手机壳截图。
10. 页面颜色、字号、按钮、标签、卡片、图标尺寸优先使用 `docs/design-foundations.md` 和 `styles/foundations.wxss`。

## 验证规则

1. `npm run wx:check` 只做安全静态检查。
2. 微信开发者工具中的“编译/预览”由人手动触发，避免脚本影响 DevTools。
3. 每个页面完成后，用真实模拟器截图对照 Figma。
4. 发现页面丑或乱，先回到规则和结构，不在坏布局上继续补丁。

## 下一步需要确认

1. 首页练手页在微信开发者工具里的实际视觉问题。
2. 首页是否继续细化交互：客户切换、识别按钮、购物车入口。
3. 下一页优先做订单列表、购物车确认页，还是客户页。
4. 数据模型：客户、商品、购物车项、订单、收款记录。
